# openwrt-docs4ai — Extended Future Roadmap (v10+)

## Where the Project Stands Today

The pipeline is **fully operational**: scraping 5 sources, validating, staging, committing, and deploying to GitHub Pages automatically. The dual-consumption architecture (complete references + individual files) is the right design for LLM tooling. The `llms.txt` indexes are well-structured and the examples index in particular is best-in-class — pedagogical ordering, pattern descriptions, file trees.

Bugs 1–2 and improvements 3–6 from the v8 pipeline review have been fixed.

---

## 🎯 Content Quality & Accuracy

### 1. Per-module function signature index
Each module's `.md` file should start with a table of contents listing every function with its signature. LLMs can scan this in seconds and jump to what they need without reading 1,500 lines. 

### 2. Semantic "when would you use this?" annotations
The current docs describe *what* each function does. Adding a one-line *when* annotation would dramatically improve relevance for LLMs answering developer questions.

### 3. Cross-link related modules
When `nl80211` docs mention `uloop.timer()`, link to the `uloop` module doc. The cross-reference system already exists for wiki docs — extend it to module docs.

### 4. Version-pinned "known working" examples
Add a curated set of tested code snippets per module that are verified against a specific OpenWrt release. Tag them with `Tested on: OpenWrt 24.10`.

## 🧑‍💻 Developer Experience (DX)

### 5. "What should I read first?" routing guide
Add a decision tree to the top-level `llms.txt` to turn the index from a "here's everything" listing into a guided experience.

### 6. Task-oriented cheat sheets
Generate condensed single-page cheat sheets for common tasks (e.g., "How to create a LuCI app from scratch", "UCI configuration patterns").

### 7. API compatibility matrix
Track which functions exist in which OpenWrt release. Even a simple table (`✅ 23.05+ / ❌ 22.03`) prevents LLMs from suggesting APIs that don't exist on the target device.

### 8. Error message catalog
Scrape common error strings from OpenWrt source code and map them to explanations + fixes. LLMs constantly get asked "what does this error mean?" — having a searchable catalog would be hugely valuable.

## 🤖 LLM Optimization

### 9. Word/line counts in per-folder indexes
Add token-budget hints to each `llms.txt` entry to let LLMs/agents decide whether a file fits in their context window before fetching it.

### 10. "Lite" versions of large references
Generate a `*-lite.md` version of massive references containing only headings, function signatures, and first-sentence summaries. ~80% size reduction while keeping navigability.

### 11. Structured metadata headers (YAML frontmatter)
Add machine-readable frontmatter to every `.md` file so LLM tools like Cursor, Continue, and Copilot can parse this for smart filtering.

### 12. Embedding-friendly chunking hints
Add `<!-- chunk -->` markers at semantic boundaries so retrieval-augmented generation (RAG) systems can split documents at meaningful points.

## 🔧 Pipeline Robustness

### 13. Incremental wiki scraping
Only re-fetch wiki pages that changed since the last run using HTTP `If-Modified-Since` headers or ETags.

### 14. File size budget tracking
Log total output size per directory and overall. Warn (or fail) if any single file exceeds a configurable threshold.

### 15. Output hash comparison
Before promoting staging → production, compare SHA256 hashes of all files. Only commit files that actually changed content (not just timestamps).

### 16. CI concurrency guard
Add `concurrency: { group: docs-pipeline, cancel-in-progress: true }` to the workflow to prevent race conditions during rapid pushes.

### 17. Scheduled monthly runs
Add a `schedule: cron: '0 6 1 * *'` trigger for automatic monthly regeneration.

## 🌟 NEW ARRIVALS: Hyper-Advanced "AI-Native DX" Ideas 

To truly push the boundaries of high-quality documentation, creativity, and modern developer experience, we must treat the LLM as a first-class developer and the pipeline as an active intelligence.

### 26. Pre-Packaged AI Prompt Snippets ("1-Click Scaffolding")
At the bottom of major reference files (e.g., `ucode-module-ubus`), inject a dynamically generated prompt template based on that file's contents. 
*Example:* `> **AI Prompt Snippet:** "I want to script a ubus listener using the ucode module described above. Please generate a skeleton script that... "` 
This drastically lowers the cognitive load for human developers trying to figure out *how* to ask the AI to use the new docs.

### 27. Semantic AST-Based "Breaking Changes" Engine
Instead of relying on textual `git diff` for `CHANGES.md`, use an Abstract Syntax Tree (AST) parser across upstream `*.c` and `*.js` source files between pipeline runs. If a function signature drops a parameter, automatically generate an incredibly rich breaking-change alert: `> [!WARNING] The signature for fs.popen() changed; the 'mode' parameter is now required.` This is far superior to standard git diffing.

### 28. Automated Code-Snippet Validation
Documentation rots when code examples break. The CI pipeline should extract every code block marked ````javascript` or ````ucode` inside the generated `.md` files and actively execute a syntax check (`ucode -c` or node linting) on them before passing the validation gate. The documentation itself becomes a tested artifact.

### 29. Mermaid.js Architecture Diagrams (Generated on the fly)
OpenWrt's architecture (procd, netifd, ubus) is heavily message-based and hard to conceptualize linearly. Use an LLM agent *during the pipeline execution* to read the scraped wiki docs and output ````mermaid` sequence diagrams, injecting them into the final markdown. This gives downstream LLMs spatial and architectural awareness when reasoning about OpenWrt event loops.

### 30. GraphRAG Knowledge Map Extraction
Flat `llms.txt` files are great for standard context windows, but modern GraphRAG systems thrive on node-edge relationships. The pipeline could output an `openwrt-knowledge-graph.jsonl` establishing relationships like `(luci-app-firewall) -[DEPENDS_ON]-> (ubus)` or `(ucode-fs) -[IMPLEMENTS]-> (POSIX I/O)`. This takes OpenWrt docs from being merely "LLM-friendly" to "GraphRAG-native."

### 31. Upstream "Documentation Preview" PR Bot
Close the feedback loop with the upstream OpenWrt/LuCI community. Build a GitHub Action that can be installed on the generic OpenWrt repos. When a developer submits a PR altering `lib/fs.c`, the bot responds with: *"Here is how your changes will appear in the `openwrt-docs4ai` LLM documentation:"*, encouraging them to write better JSDoc comments directly at the source.

### 32. "Doc Chats" (Few-Shot Conversational Tuning Data)
LLMs learn best by example. Append a curated file `openwrt-doc-chats.md` containing 10 simulated conversations between a "Senior OpenWrt Dev" and a "Junior Dev". 
*Junior: "How do I securely pass a password to a ucode popen process?"*
*Senior: "You should never pass it as an argument; supply it via stdin instead. Here is how... "*
Feeding this directly via `llms.txt` acts as instant instruction-tuning for any chat model, dramatically raising the quality of its subsequent answers.

### 33. Pipeline Self-Healing (Resilience Layer)
If `openwrt.org` changes their DokuWiki HTML theme, `02a-scrape-wiki.py` will break. Instead of immediately failing the CI run, the pipeline hooks into the GitHub Models API, feeds the broken HTML + the Python script to the LLM, asks for a regex/BeautifulSoup patch, and automatically commits the patch to a `fix/scraper-auto-heal` branch for you to review. The pipeline writes its own maintenance code.

### 34. Auto-Generate Project `AGENTS.md` Files
`AGENTS.md` is an emerging open standard serving as a "README for AI agents" (used by Cursor, Copilot, and Claude Desktop). It teaches agents about project conventions, build commands, and testing protocols. The pipeline could synthesize an `openwrt-condensed-docs/AGENTS.md` file specifically designed to instruct local AI tools on how to write OpenWrt C code or LuCI JS. 

### 35. Dual-Faceted Indexing (`llms-full.txt` standard)
The formal `llms.txt` web specification proposes two files: `/llms.txt` for a streamlined navigational overview, and `/llms-full.txt` containing the comprehensive, concatenated documentation. We should formalize our `*-complete-reference.md` files by aggregating them into an official `llms-full.txt`. This allows zero-config ingest by external AI aggregation platforms (like Mintlify or Docling) that look for this specific filename.

### 36. Q&A Formatted Synthetic Data (RAG Optimization)
In the pipeline's Step 04 (AI Summaries), we currently generate paragraph summaries. We should pivot this to generate 5 "Frequently Asked Questions" for every module. Documentation written in Q&A format mathematically improves Retrieval-Augmented Generation (RAG) vector embeddings and chunking relevance compared to standard declarative prose.

### 38. Workflow Push Trigger Optimization
Currently, the CI pipeline is configured to trigger `on: push: branches: [main]`. This means when the pipeline itself pushes the updated documentation to the `openwrt-condensed-docs/` folder, it could technically trigger *another* full 45-minute run (though GitHub Actions tries to prevent infinite loops from `github-actions[bot]`). More importantly, when a human pushes a simple roadmap update to `docs/plans/`, the entire scraping pipeline runs unnecessarily. We should restrict the `push` trigger using `paths: ['.github/**']` or remove the push trigger entirely now that the pipeline is stable, relying solely on the monthly `schedule` and manual `workflow_dispatch`.
