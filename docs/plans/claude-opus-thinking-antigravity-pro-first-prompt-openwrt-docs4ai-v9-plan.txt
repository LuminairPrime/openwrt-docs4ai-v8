one developer's suggestion:

A solid, low-risk pattern on GitHub Actions is: generate everything under $RUNNER_TEMP, run smoke tests, then copy only the verified outputs into $GITHUB_WORKSPACE (the checked-out repo) and commit/push.

How Actions runner disk works
GitHub-hosted runners give you an ephemeral machine for the job, and your repository is checked out under GITHUB_WORKSPACE (example path: /home/runner/work/<repo>/<repo>). GitHub also provides RUNNER_TEMP, “a temporary directory on the runner” that is emptied at the beginning and end of each job, which is exactly what you want for “don’t overwrite the good committed files until validated.” GitHub explicitly recommends using these variables rather than hardcoded paths, and notes you can’t override default GITHUB_*/RUNNER_* variables.

What other people do
A common approach is “generate into an output folder, then commit that folder at the end,” which shows up in GitHub Community examples. Another common approach is to commit only at the end using an action like stefanzweifel/git-auto-commit-action (or similar), so your workflow logic focuses on generate/test and the “commit plumbing” is standardized. If you don’t actually want to commit results yet, teams often stage outputs and upload them as workflow artifacts instead (so you can inspect/download them without touching the repo).

Detailed migration plan
Define three directories (don’t blur them together): WORKDIR=$RUNNER_TEMP/work, OUTDIR=$RUNNER_TEMP/out, REPO=$GITHUB_WORKSPACE.

Refactor scripts so they never assume “repo root” for outputs; instead, each script takes --out-dir (or OUTDIR env var) and writes only under that directory.

Run smoke tests against OUTDIR (existence checks, parse checks, invariants like “non-empty”, schema/required headings), and fail fast on any error (non-zero exit).

Only after tests pass, “promote” by copying from OUTDIR into REPO (ideally into a dedicated publish folder like docs/ or public/ rather than repo root), then git add only those known paths and commit/push.

On failure, upload OUTDIR as an artifact for debugging (so you can compare “bad output” vs “last good output” without ever overwriting the repo).

Pseudocode (workflow + scripts)
Workflow skeleton (pseudocode-ish YAML):

Set OUTDIR: ${{ runner.temp }}/out and WORKDIR: ${{ runner.temp }}/work so every step uses runner temp storage.

Steps: checkout → generate to OUTDIR → smoke-test OUTDIR → promote to $GITHUB_WORKSPACE/<publish_dir> → commit/push (or open PR) → upload artifacts on failure.

Script skeleton:

text
out_dir = arg("--out-dir") or env("OUTDIR")
tmp_file = out_dir + "/file.ext.tmp"
final_file = out_dir + "/file.ext"

write(tmp_file)
if not smoke_check(tmp_file): exit(1)
atomic_rename(tmp_file, final_file)   # ensures partial writes don’t “look valid”
exit(0)
“Promotion” safety checks
Before committing, run git diff --stat (or git status --porcelain) and abort if changes touch unexpected paths; this prevents accidental mass overwrites. If you do automated commits, prefer committing only after tests pass and avoid broad “sync-to-root” operations that can delete unrelated files (instead, copy only the known generated outputs into a known destination).
